SwEng 2014 Final Exam¶

In this final exam, you will add functionality to an existing piece of code, using the software engineering practices you learned in this class. You will be working on a code base that provides operations on binary trees. Initially, the operations are basic — iterating over the elements of the tree. Then, the code base evolves; the tree gets the ability to represent simple mathematical expressions, and then an interpreter and expression evaluator are added.

The goal of this exam is to enable you to prove:

  • You understand version control systems (git) and coding workflow.
  • You understand the specification of an interface and can implement it correctly.
  • You understand the design patterns seen in lecture and can implement them.
  • You are able to test code.
  • You are able to debug code.

This practical part of the exam is worth 80 points in total. Try to get as many as possible, but don’t panic if you get less than 80.

There are three tasks in this part the exam, of varying difficulty. The three tasks are independent; if you get stuck on one, you can switch to the others.

WARNING: Most of this exam will be graded automatically. The grading scripts are very pedantic. Please be careful not to modify the tests that we provided in your repository, and make sure that the tests can still compile and run on the version of the code you hand in.

Code that does not test properly because of a naming error will be marked as incorrect.

Commit early and often!¶

Commit and push frequently, ideally after every task. This ensures that you will get points for all tasks you complete.

Merge branches to master!¶

The problem statement requires you to write code in various branches of the repository. After implementing a feature, you are required to merge the branch into master. Be careful not to break the master branch! You will still receive points if you only push changes to the branches, without merging, but for maximum points, correctly functioning code for each problem needs to be merged into master by the end of the exam.

Push all branches to Github!¶

Make sure that when you commit on a certain branch, you also push that branch to Github, not only the master branch. For example, if you make changes in the iterator branch, make sure you execute git push origin iterator (or an equivalent command).

Problem Statement¶

Task #1 (20 points)¶

Switch to the iterator branch in the repo. It implements an in-order iterator for the tree. The iterator has a single bug, where it violates the Iterator specification.

  • Write a test to show the bug. (A new test in ch.epfl.sweng.test.InOrderIteratorTest)
  • Commit the test that shows the bug.
  • Fix the bug.
  • Commit the fix.
  • Make sure that your test passes on your fixed code and that the existing tests of the iterator still pass.
  • Push the iterator branch to Github.
  • Merge the branch into master.
  • Push the master branch to Github.

Task #2 (30 points)¶

Switch to the evaluator branch in the repo. It has a framework that allows you to implement a simple expression evaluator using the visitor pattern. Due to the missing code, the code on the branch does not compile yet and the existing test currently fails. The evaluator should throw an IllegalArgumentException if the expression tree is invalid (if AddNode or MultiplyNode do not have exactly two children).

  • Fill in the missing parts of the evaluator. You can write additional private methods if you want, but do not change the signatures of methods we provided.
  • Commit the code.
  • Write one new test that verifies that the evaluator correctly evaluates a valid expression tree. (A new test in ch.epfl.sweng.test.EvalExpressionVisitorTest)
  • Write one new test that verifies that the evaluator correctly throws an IllegalArgumentException for invalid expression trees. (A new test in ch.epfl.sweng.test.EvalExpressionVisitorTest)
  • Commit your tests.
  • Make sure that your tests pass and that our test also passes.
  • Push the evaluator branch to Github.
  • Merge the branch into master.
  • Push the master branch to Github.

Task #3 (30 points)¶

Switch to the interpreter branch in the repo. It has the implementation of a simple mathematical expression interpreter supporting addition and multiplication.

The interpreter has two bugs. It accepts expressions that are not valid (it does not throw InvalidSyntaxException, even though it should). The InterpreterTest has one test that highlights one of the bugs. It also has a test that checks that the interpreter correctly parses a valid expression.

  • Fix the bug highlighted by the test that currently fails.
  • Commit the code.
  • Write an additional test, highlighting the other bug. (A new test in ch.epfl.sweng.test.InterpreterTest)
  • Commit the code.
  • Fix the second bug.
  • Commit the code.
  • Make sure that both tests we provided pass.
  • Push the interpreter branch to Github.
  • Merge the branch into master.
  • Push the master branch to Github.

Hint. You do not need to fully understand the interpreter in order to fix the bugs. Try to use the debugging tools at hand (the Eclipse debugger, printing) to understand the test executions that highlight the bug. The simpler the test, the shorter and cleaner the execution. You can call tokenizer.toString() to see the current element that is being looked at by the interpreter. You can either print tokenizer.toString(), or you can inspect it in the Expressions debug window in Eclipse (Window->Show View->Expressions).
